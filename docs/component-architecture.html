Responsibility split

Where each kind of decision lives

The architecture is small, but responsibility crosses three different state mechanisms: SwiftUI state, SwiftData observation, and controller-mediated writes.

UI

Views

Own presentation state, sheets, alerts, filters, sorting, queries, and platform-facing actions such as sharing and browser launch.

VM

Editor view model

Copies a model into editable fields, guards the hidden toggle, calculates editor height, and chooses add versus update at save time.

CT

MainController

Provides one mutation surface for links and groups, enforces HTTP(S) on browser opening, and delegates URL metadata operations.

SV

Services

Isolate network parsing, authentication, and QR generation from the data model and most presentation code.

DM

Domain models

Store persistent fields and enforce update-time rules, including clearing groups whenever a link becomes hidden.

DB

Persistence

SwiftData publishes query changes to views. UserDefaults separately stores display preferences and hidden-feature availability.

Architectural seam: createLink() and createNewGroup() deliberately return unsaved models. The editor can be cancelled without inserting them; insertion happens only when the user saves.
