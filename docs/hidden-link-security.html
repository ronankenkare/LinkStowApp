06 · Security sequence

Three entry points share one trust decision.

Enabling the hidden feature, viewing hidden records, and marking a link hidden all rely on the device-owner authentication policy supplied by iOS.

LocalAuthenticationUI access controlLocal data

Hidden-link authentication sequence

Read downward. iOS owns credential collection and evaluation; LinkStow receives a boolean result and applies the corresponding UI or model transition.

    opt Enable the hidden-group feature
        User->>Pref: Turn Hidden Group on
        Pref->>Auth: authenticateUser(activation: true)
        Auth->>IOS: canEvaluatePolicy(deviceOwnerAuthentication)
        IOS-->>Auth: Capability available or unavailable
        alt Policy can be evaluated
            Auth->>IOS: evaluatePolicy(reason)
            IOS-->>Auth: Success, cancel, or error
        end
        alt Authentication succeeds
            Auth-->>Pref: true
            Pref->>Defaults: hiddenGroupActive = true
            Defaults-->>Main: Hidden bubble becomes available
        else Authentication fails
            Pref->>Pref: Revert toggle to previous value
        end
    end

    opt Access hidden records
        User->>Main: Tap locked Hidden bubble
        Main->>Auth: authenticateUser(activation: false)
        Auth->>IOS: evaluate device-owner policy
        IOS-->>Auth: Result
        alt Success
            Auth-->>Main: true
            Main->>Main: filterHiddenSelected = true
            Data-->>Main: Supply hiddenLinks @Query
            Main-->>User: Render hidden-link list
        else Failure, cancel, or error
            Main->>Main: isAuthenticating = false
            Main-->>User: Keep hidden filter closed
        end
    end

    opt Mark a visible link hidden
        User->>VM: Toggle Hidden in editor
        VM->>Auth: authenticateUser(activation: false)
        Auth->>IOS: evaluate device-owner policy
        IOS-->>Auth: Result
        alt Success
            VM->>VM: isHidden = true
            User->>VM: Save editor
            VM->>Model: updateLink(isHidden: true)
            Model->>Model: Clear groups
            Model->>Data: Persist hidden state
        else Failure
            VM->>VM: Leave isHidden false
        end
    end

    opt Disable the hidden-group feature
        User->>Pref: Turn Hidden Group off
        Pref->>Defaults: hiddenGroupActive = false
        Note over Pref,Defaults: Disabling does not request authentication
    end

Security interpretation

Control, asset, boundary, and limitation

This view distinguishes what the code demonstrably protects from protections that are not implemented or asserted.

A

Protected asset

The app protects access to hidden-link UI paths and requires authentication before converting a visible editor draft into hidden state.

B

Trust boundary

Credential entry and biometric/passcode evaluation stay inside iOS. LinkStow supplies only a reason string and receives the policy result.

C

Control behavior

Failure leaves the prior state unchanged: preferences revert, the hidden filter stays closed, or the editor remains visible-state.

L

Current limitation

The code does not define a separately encrypted hidden-link database. The control is application-level authentication and concealment over local SwiftData.

As-built security note: turning the hidden feature off requires no authentication, and the source does not describe data-at-rest encryption beyond protections provided by the device and SwiftData environment.
